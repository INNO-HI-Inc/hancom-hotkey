What is Hancom Office?

Hancom Office (HWP / Hancell / Hanshow) is the de-facto Korean office suite, used by virtually every Korean government agency, university, and large enterprise. The Mac version uses Ctrl-based shortcuts that feel unnatural on macOS — HancomHotkey fixes that.

If ⌘0–4 do nothing — 5-step HWP style setup

Hancom Mac auto-binds shortcuts by style list position: 1st = ⌃ 1 (Body), 2nd = ⌃ 2, etc. This app translates ⌘ 0–4 to ⌃ 1–5, so you need meaningful styles in slots 1–5 for anything to happen.

  1. Open the Styles panel — HWP menu: Format → Styles… (or F6; on Touch Bar Macs use fn + F6)
  2. Select the style to move — click the style you use often
  3. Move with arrow buttons — use the ▲▼ buttons on the right, or drag
  4. Place in slots 1–5 — ⌘ 0 = slot 1 (Body), ⌘ 1 = slot 2, …, ⌘ 4 = slot 5
  5. Done — closing the panel auto-saves. Try ⌘ 0–4 in HWP to confirm

The same guide is also available in the app's Settings → Shortcuts tab.

Why does it need Accessibility permission?

▲ Key remapping requires macOS Accessibility permission

Detecting ⌘ 0 and substituting ⌃ 1 is technically "key hooking" on macOS, which requires Accessibility privileges.

  • Only intercepts keys when HWP / Hancell / Hanshow is frontmost
  • Auto-disables in Secure Input contexts (passwords, lock screens)
  • No keystrokes are logged, stored, or transmitted
  • Open source + Apple Developer ID signed + notarized

Path: System Settings → Privacy & Security → Accessibility → toggle on HancomHotkey. The first-run wizard opens this for you.

FAQ

First press only changes alignment; I have to press twice for the font to change

This is HWP's intended behavior, not a bug in this app. Per Hancom's official documentation:

  • First application: applies the new style but preserves any character/paragraph formatting you manually changed.
  • Second application of the same style: HWP shows "Overwrite with style?" or auto-overrides — your manual edits get reset to the style definition.

So if you previously hand-edited the font on that paragraph, you need two presses to fully apply the style. Workaround: hit ⌃ 1 (Body) to reset first, or apply to a fresh paragraph. Source: Hancom official docs — Applying styles.

I press ⌘0–4 but nothing happens

The most common cause: HWP style list slots 1–5 are empty or contain auto-generated styles with no visual change.

Hancom Mac auto-binds ⌃ 1–⌃ 0 by style list position. This app only translates ⌘ 0–4 to ⌃ 1–5 — if those slots have no meaningful style, nothing changes.

Fix: follow the 5-step setup above to move your favorite styles into slots 1–5.

Other things to verify:

  • Menu bar status reads ● Active (not paused or "permission needed")
  • Accessibility permission is granted (System Settings → Privacy & Security → Accessibility)
  • The frontmost window really is HWP / Hancell / Hanshow
Can I customize the key bindings?

v0.4.0 ships with a fixed mapping (⌘ 0–4 → ⌃ 1–5). Custom mappings are under consideration — please file a request on GitHub Issues.

Does it interfere with other apps' shortcuts?

No. Remapping only fires when an HWP / Hancell / Hanshow window is frontmost. In Safari, Slack, Xcode, etc., ⌘ 0–4 behaves normally. You can also pause everything with ⌃⌥⌘P.

Can it launch at login?

Yes. Menu bar → Settings → General → enable "Launch at login". Uses macOS SMAppService, no extra permission needed.

Are my keystrokes safe? What about passwords?

Yes. macOS triggers Secure Input mode in password fields (banking, 1Password, lock screen, etc.), and HancomHotkey detects this and auto-disables remapping.

The app never logs or transmits keystrokes — it intercepts ⌘0–4, sends ⌃1–5 instead, and discards from memory. The full source is on GitHub for verification.

How do I uninstall?

1) Menu bar → Quit · 2) Drag HancomHotkey.app to Trash · 3) Remove from System Settings → Accessibility · 4) Optional preference cleanup: defaults delete com.innohi.hancomhotkey
